# wechat-copilot · 需求与技术选型

> 自用 AI 微信恋爱军师。本文件 = 需求事实 + 技术选型决策，随讨论更新。

## 一、需求（2026-06-06 对齐）

**目标**：跟相亲女生微信聊天时，给我「分析 + 能发的草稿」，推进到约会见面。**教练定位**，重在提升我自己。

**形态**：**手机为主**。既要「实时救场」（正聊着卡住，当场要下一句）+「事后复盘谋划」，两者都要。

**输入**（我喂给它的）：
- 她的消息 + 我的消息（**双方原文**）
- 我的**意图 / 画外音**：随时补，比如"想周末约她吃饭、划船"、对她的判断、我的顾虑

**输出**（它给我的）：
- **分析**（她在想啥 / 阶段 / 异性视角）
- **能直接发的草稿**（2–3 条带策略）
- 两者都要

**范围**：多对象，**按人分离**（不同人各一份数据，互不串）。

**连续性**：聊得越多越要"**能继续接话**"——长期记住每个人，老对话压缩成摘要，不能每次从零。

## 二、存储设计（建议）

每个对象一个目录，用户隔离：

```
data/<对象>/
  messages.jsonl   # 原文,append-only,真相源   {ts, who:我/她, type, text}
  intents.jsonl    # 我的意图/画外音,带时间      {ts, note}
  turns.jsonl      # 每次咨询的输出             {ts, analysis, drafts:[...]}
  profile.md       # 慢变档案: 她是谁/喜好/雷区/关系阶段/邀约记录
  summary.md       # 滚动摘要(老对话压缩) → 保证"继续接话"
```

要点：
- **原文用 JSONL**（结构化、只追加、是真相源，对应你说的"原文很重要"）。
- **意图/画外音单独存**（不污染真实对话，但带时间能跟对话对齐）。
- **分析 + 草稿存同一条 turn 记录、字段分开**（既能整体回看，又能各取所需）——回答你"是不是分别存"：同一条、分字段。
- **profile + summary = 连续性层**，老对话压进 summary，profile 留长期事实。

## 三、技术选型

**先砍掉的**：
- **Claude Code 当主运行时** ❌ —— 它是桌面交互工具，**手机够不着**（你形态=手机为主）。仍可用来开发/迭代，但不是日用运行时。
- **LangChain** ❌ —— 你的活是"拼上下文调一次模型"，它的抽象层纯负担，单人 debug 到吐。
- **OpenClaw** ❌ —— 瞄准自主体(Tier C)，为"单次调用"养 24h 运行时，维护+安全面对单人是灾难。
- **工具调用（自主 tool-loop）** ❌ —— 不需要；取数据/写档案用确定性代码直接调。

**真正的两条路**：

1. **（推荐）自托管薄服务 + IM 机器人界面**
   - 一个小服务（FastAPI），手机用**企业微信 / 飞书机器人**跟它对话：把双方消息 + 意图发进去 → 回分析 + 草稿。
   - 跑在哪：**本机(Mac) + reverse-http-tunnel 暴露给手机**（复用你现成隧道，呼应"本机运行就行"，但 Mac 得开着）；或 **CVM**（always-on，实时救场更稳）。
   - 模型：Qwen3-Max（DashScope，快/便宜/中文）。存储=上面那套你自己定的格式。复用 `coach.py` 内核。
   - 优点：控制力、隐私、存储格式你说了算，契合"原文很重要 + 按人分离"。

2. **自托管 Dify 智能体**
   - 书当知识库、prompt 填进去、发布到企业微信/飞书，**少写代码**。
   - 代价：存储格式不由你定、深定制别扭、"原文单独结构化存"难落地。

**待拍板（2 个）**：
- **界面渠道**：企业微信 / 飞书 / 网页？（你提过企业微信转发没问题）
- **模型**：Qwen3-Max（默认）还是 Claude API（质量更高，但北京→Anthropic 有网络/成本坑）？
